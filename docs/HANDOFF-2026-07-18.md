## 3. Analytical findings (revisions to the incident report)

1. **The 13:25 outage was likely a self-reboot, not a human power cycle.**
   Last log line of old boot: 13:25:30 (NTP-corrected). True new-boot time
   per `uptime -s`: 13:25:59. A **29-second gap** ‚Äî nobody notices a dark
   rig and power-cycles two devices in 29 s. If the hardware watchdog was
   armed (harden.sh), a kernel hang ‚Üí forced reboot leaves *exactly* this
   signature (log stops mid-poll, no shutdown trail, back in ~15‚Äì30 s).
   "No shutdown sequence" is the signature of power loss **or**
   hang/watchdog-reset ‚Äî the report treated it as diagnostic of power
   loss; it isn't.
2. **"Rig went fully dark" is explained by the reboot itself:** fresh boot
   initializes all 32 channels to 0 (blackout) ‚Äî logged at reboot. Fixtures
   can be fully powered and dark. So the dark rig does not prove fixture
   power loss; the venue's power-cycling probably happened *after* the Pi
   had already self-recovered.
3. **The "not software" flicker verdict is weaker than stated.**
   `dmx-monitor` polls `/api/channel_values` = QLC+'s *intended* output,
   not the wire. FTDI/USB dropouts (classic undervoltage symptom on Pi 3)
   flicker the fixtures while the log stays flat. Also 1 Hz sampling
   aliases any oscillation faster than ~0.5 Hz. The negative result
   exonerates scenes/chases/control-layer only ‚Äî **not** the USB‚Üíwidget‚ÜíDMX
   path.
4. **One root cause can explain everything:** marginal 5 V supply to the
   Pi ‚áí USB brownouts (flicker) escalating to a hang (outage) ‚áí watchdog
   reboot ‚áí blackout. This is testable remotely ‚Äî nothing in the repo
   currently reads `vcgencmd get_throttled` (verified by grep).
5. **The wifi-watchdog journal is an accidental router-uptime monitor**
   (OK/FAIL every 2 min). It can settle whether the router actually went
   down at 13:25 (‚áí shared power sag at the outlet) or was fine (‚áí its
   power-cycle was precautionary).
6. Additional causations the report didn't list: SlimPar sound-active/auto
   fallback on DMX loss (fits "both fixtures, blue-ish flashes"), DMX
   personality/address mismatch, PWM/dimmer-curve flicker at low dim
   levels, aging SD card (stall ‚Üí hang ‚Üí watchdog reboot).

## 6. Phase 2 ‚Äî Remote diagnostics on the Pi (`./lightsctl.sh ssh`)

Run all; paste results back to the user with interpretation.

```bash
# (a) THE decisive check ‚Äî undervoltage/throttle flags
vcgencmd get_throttled
#   0x0 = clean. Bit 0/16 undervoltage now/since-boot; 2/18 throttled now/since-boot.
#   e.g. 0x50005 ‚áí PSU/cable implicated ‚áí flicker+hang likely power-rooted.

# (b) Kernel evidence, current + crashed boot
journalctl -k -b 0  | grep -iE 'under-?voltage|usb 1-|ftdi|mmc0|i/o error'
journalctl -k -b -1 | grep -iE 'under-?voltage|usb 1-|ftdi|mmc0|i/o error'
#   FTDI disconnect/reconnect during 12:00‚Äì13:25 Jul 18 = the flicker, found.

# (c) Reboot history ‚Äî chronic vs one-off
journalctl --list-boots
last -x reboot shutdown | head -20

# (d) Was the hardware watchdog armed? (decides the ¬ß3.1 story)
systemctl show -p RuntimeWatchdogUSec
grep -h watchdog /boot/config.txt /boot/firmware/config.txt 2>/dev/null
ls -l /dev/watchdog*
#   Not armed? Run `./lightsctl.sh harden` + reboot (coordinate with user ‚Äî reboot!).

# (e) SD card health
dmesg -T | grep -iE 'mmc|card'

# (f) Router uptime at incident time ‚Äî the accidental monitor
journalctl -t wifi-watchdog --since "2026-07-18 12:00" --until "2026-07-18 14:00"
#   FAILs before 13:25 ‚áí router died too ‚áí shared power sag.
#   OK throughout ‚áí router fine, its power-cycle was precautionary.

# (g) Close the loose thread: the 12:24:24/12:25:13 all-channel resets
journalctl -u lighting-control.service --since "2026-07-18 12:23" --until "2026-07-18 12:26"
#   Look for a workspace/scene-reload request matching both timestamps.
```

## 7. Phase 3 ‚Äî Code mitigation backlog (discuss with user before building)

   look instead of blackout (turns a self-reboot into a blip).
5. Physical (user, on site): official 5.1 V/2.5 A PSU if (a) shows
   undervoltage; DMX terminator + real DMX cable on the SlimPar run; check
   SlimPar menus (DMX mode/channel count, address, dimmer curve, disable
   sound-active fallback); fresh A1 SD card if (e) shows mmc errors; RTC
   module (~$10); UPS only if power is confirmed as the cause.

## 8. Repo conventions that bit us / will bite you

- `CLAUDE.md` is authoritative: conventional commits (`feat(scope):` ‚Ä¶),
  **no `Co-Authored-By` trailers**, **no `claude` in branch names** (hence
  the rename in Phase 1), **never push to `main`** on lights-pi (PRs only;
  `lights-pi-www` commits straight to `main`).
- `cd control-server && python -m pytest -q` before committing (expect the
  3 known `test_tempo` flakes locally ‚Äî see ¬ß4).
- Deploy = `bash scripts/deploy.sh` (rsync + service restart). `.env` never
  syncs ‚Äî Pi secrets are edited on the Pi. Hard-refresh the browser after
  deploys.
- `lightsctl.sh update` is OS packages only ‚Äî it is not a code deploy.
